### What is x402

The HTTP error code **402 Payment Required** has been in the original specification of the HTTP protocol since the 1990s and is described as being reserved for future use.

In 2025, the Coinbase developer platform has picked up the implementation of this error code and created the **x402 protocol**, an emerging **internet-native payment standard** at the HTTP layer. It enables novel payment methods such as micropayments, pay-per-API requests, and machine-to-machine payments.

### Common x402 use cases

*Pay-per-API access:* Charge per request without managing plans or quotas.

*Agent-to-service payments:* Allow AI agents to discover services and pay for usage autonomously.

*On-demand compute and inference:* Meter and charge for execution time or inference requests.

*Premium data and content APIs:* Sell access to high-value data without subscriptions.

*Internal service monetization:* Enable usage-based billing and cost attribution across microservices.

### x402 on Algorand

Although x402 does not require a blockchain, it still gains from using a blockchain as payment rails.
On Algorand, that shows up in capabilities built into the protocol itself:

- Predictable performance
- Low transaction fees
- Instant finality
- Stablecoin support
- Atomic transaction groups
- High throughput

### Architecture and roles

There are different roles in a typical x402 scenario.

- **Merchant** — Wants to provide a resource. They decide on pricing, access rules, and payment terms.
- **Resource server** — Protects access to these resources and enforces the x402 requirements.
- **Facilitator** — Verifies payments and checks settlements (often coordinating atomic transaction groups on Algorand).
- **Client** — Such as AI agents or backends, requesting an x402-protected resource over HTTP, MCP, A2A, or other supported transports.

### x402 request flow

A typical x402 interaction follows this pattern:

1. A client or agent requests a protected resource.
2. The server responds with HTTP 402 and structured payment requirements.
3. The client prepares payment and submits payment proof.
4. The facilitator verifies settlement.
5. The server returns the requested resource.

## Build a pay-per-API service with x402

The quickest way to get started is by implementing an x402-enabled **client** that talks to an x402 resource server, then building your own **resource server** and wiring it to a **hosted facilitator** (GoPlausible in this walkthrough).

You will use a simple **`/weather`** API: unpaid calls get `402 Payment Required`, and after payment the client gets JSON back.

### Setup

#### What you will need

**Node.js** and a TypeScript-friendly setup. Use a current **Node.js** release. **npm** ships with Node.js; you use it to install packages and **`npx tsx`** to run **TypeScript** files.

Follow these steps **in order** so balances and opt-ins are ready even before you have to run any code:

1. **Create two Algorand TestNet accounts** — one for the **client** (pays) and one for the **resource server** (receives payments). If you need a walkthrough, see [create an account](/concepts/accounts/create/).

2. **Fund both with TestNet ALGO** using [Lora TestNet fund](https://lora.algokit.io/testnet/fund). You need ALGO for transaction fees and for the minimum balance when you hold USDC. More context: [fund an account](/concepts/accounts/funding/).

3. **Opt both accounts in to TestNet USDC** ([ASA 10458941 on Lora](https://lora.algokit.io/testnet/asset/10458941)). Until you opt in, you cannot hold or send that asset — see [opting in and out of assets](/concepts/assets/asset-operations/#opting-in-and-out-of-assets).

4. **Add TestNet USDC** to **both** addresses using [Circle’s testnet faucet](https://faucet.circle.com/) (pick **Algorand Testnet**, paste each address; rate limits are described on that page). The client spends USDC; the server account receives USDC when you run your own monetized route.

5. **Save mnemonics and addresses securely** for the next steps: the client project uses the payer mnemonic as `AVM_MNEMONIC`; the server project uses the recipient **public address** as `AVM_ADDRESS`.

#### Add x402 payment support to the client

**Replace** the contents of `index.ts` with the version below. The comments mark the main steps: initialize the x402 client, create the Algorand signer, register the exact AVM scheme, and wrap `fetch` with payment handling.

```typescript
import { config } from "dotenv";
import { x402Client, wrapFetchWithPayment, x402HTTPClient } from "@x402-avm/fetch";
import { toClientAvmSigner } from "@x402-avm/avm";
import { registerExactAvmScheme } from "@x402-avm/avm/exact/client";
import { ed25519SigningKeyFromWrappedSecret, type WrappedEd25519Seed } from "@algorandfoundation/algokit-utils/crypto";
import { seedFromMnemonic } from "@algorandfoundation/algokit-utils/algo25";

config();

const avmMnemonic = process.env.AVM_MNEMONIC as string;
const url = "https://x402.goplausible.xyz/examples/weather";

async function main(): Promise<void> {
    const secretKey = await getSecretKeyFromMnemonic(avmMnemonic);
    // Create the Algorand signer used to authorize payments.
    const avmSigner = toClientAvmSigner(secretKey);

    // Initialize the x402 client.
    const client = new x402Client();
    // Register the exact AVM scheme so the client knows how to satisfy Algorand "exact" requirements.
    registerExactAvmScheme(client, { signer: avmSigner });
    console.info(`AVM signer: ${avmSigner.address}`);

    // Wrap fetch so 402 responses trigger payment handling automatically.
    const fetchWithPayment = wrapFetchWithPayment(fetch, client);

    const response = await fetchWithPayment(url, { method: "GET" });

    if (response.ok) {
        const paymentResponse = new x402HTTPClient(client).getPaymentSettleResponse(name =>
            response.headers.get(name),
        );
        console.log("\nPayment response:", JSON.stringify(paymentResponse, null, 2));
    } else {
        console.log(`\nNo payment settled (response status: ${response.status})`);
    }
}

async function getSecretKeyFromMnemonic(avmMnemonic: string): Promise<string> {
    const seed = seedFromMnemonic(avmMnemonic);
    const seedCopy = new Uint8Array(seed);
    const wrappedSeed: WrappedEd25519Seed = {
        unwrapEd25519Seed: async () => seed,
        wrapEd25519Seed: async () => { },
    }
    const wrappedSecret = await ed25519SigningKeyFromWrappedSecret(wrappedSeed)
    return Buffer.concat([
        Buffer.from(seedCopy),
        Buffer.from(wrappedSecret.ed25519Pubkey),
    ]).toString('base64');
}

main().catch(error => {
    console.error(error?.response?.data?.error ?? error);
    process.exit(1);
});

```

The function `registerExactAvmScheme()` lets the `x402Client` pay on Algorand using the **exact** scheme from the [x402 specification](https://github.com/coinbase/x402).

Your payer account must be funded, opted in, and holding TestNet USDC as described in [Setup](#setup) before this will succeed.

#### Execute the payment flow

Run `npx tsx index.ts` with `AVM_MNEMONIC` set and the client ready. You should see payment requirements handled automatically: the client prepares payment, submits proof through the wrapped `fetch`, and receives the protected weather response once the facilitator and server settle.

You have now executed your very first x402 payment!

The layout matches the **`client/fetch`** tree linked in **Client setup**. Browse [`client/`](https://github.com/algorand-devrel/x402-demo/tree/main/client) for other client samples in the same repo.

#### Under the hood

From the decoded instructions, you pay **10000** units of ASA **10458941** (TestNet USDC, **6** decimals) — **$0.01** — to the listed address. That matches the **GoPlausible weather example** used in Part 1; it is not tied to the price you choose for your own server in Part 2.

When a `feePayer` is set, that account can cover transaction fees so you mainly move USDC.

The `exactAvmScheme` path builds an [atomic transaction group](/concepts/transactions/atomic-txn-groups), signs the client’s leg, and sends proof to the resource server. The server forwards to its facilitator, which may co-sign (for example fee sponsorship), submits the group, and reports back. The server then returns the resource or a **corresponding** error to the client.

### Part 2: Build the resource server

#### Resource server setup

What about the other side? As the merchant, you want to see how to provide an x402-protected resource.

You do that by setting up a simple TypeScript project, as above, and adding a [Hono](https://hono.dev/) HTTP server.

```bash
mkdir x402-demo-server
cd x402-demo-server
npm init -y
npm install -D typescript @types/node tsx
npx tsc --init
npm install @x402-avm/hono@^2.6.1 @x402-avm/avm@^2.6.1 @x402-avm/core@^2.6.1 hono @hono/node-server dotenv
```

Versions align with [x402-demo server (Hono)](https://github.com/algorand-devrel/x402-demo/tree/main/server/hono).

Apply the same `package.json` `"type": "module"` and `tsconfig.json` `"types": ["node"]` adjustments you used for the client.

:::note
**Reminder:** create **`.env`** and **`.gitignore`** in this project root if you do not have them yet. List **`.env`** in **`.gitignore`**.
:::

This time, you provide the account address in the `.env` file and the facilitator endpoint. This walkthrough uses GoPlausible’s facilitator: `https://facilitator.goplausible.xyz`.

```
# Resource server account — use its public 58-character Algorand address (not the mnemonic)
AVM_ADDRESS=

FACILITATOR_URL=https://facilitator.goplausible.xyz
```

Fill in **`AVM_ADDRESS`** with the TestNet public address for the **resource server** account from [Setup](#setup).

Again, you use the `@x402-avm` libraries that take care of much of the heavy lifting.

#### Create `index.ts` and protect `/weather`

In the server project root, create **`index.ts`**. The listing wires Hono, the facilitator client, and `paymentMiddleware` so `GET /weather` is paid first. The `accepts` array states the **scheme** (`exact`), **price**, **network**, and **payTo** (your `AVM_ADDRESS`). Here **`price: "$0.001"`** is what **you** charge on this route; it does not need to match the **$0.01** per request shown in the GoPlausible decoded payload from Part 1.

```typescript
import { config } from "dotenv";
import { paymentMiddleware, x402ResourceServer } from "@x402-avm/hono";
import { ExactAvmScheme } from "@x402-avm/avm/exact/server";
import { HTTPFacilitatorClient } from "@x402-avm/core/server";
import { Hono } from "hono";
import { serve } from "@hono/node-server";
config();

const avmAddress = process.env.AVM_ADDRESS;
if (!avmAddress) {
    console.error("Missing AVM_ADDRESS environment variable");
    process.exit(1);
}

const facilitatorUrl = process.env.FACILITATOR_URL;
if (!facilitatorUrl) {
    console.error("Missing FACILITATOR_URL environment variable");
    process.exit(1);
}
const facilitatorClient = new HTTPFacilitatorClient({ url: facilitatorUrl });

const accepts = [
    {
        scheme: "exact",
        price: "$0.001",
        network: "algorand:SGO1GKSzyE7IEPItTxCByw9x8FmnrCDexi9/cOUJOiI=" as const,
        payTo: avmAddress,
    },
];

const server = new x402ResourceServer(facilitatorClient)
    .register("algorand:SGO1GKSzyE7IEPItTxCByw9x8FmnrCDexi9/cOUJOiI=", new ExactAvmScheme());

const app = new Hono();

app.use(
    paymentMiddleware(
        {
            "GET /weather": {
                accepts,
                description: "Weather data",
                mimeType: "application/json",
            },
        },
        server,
    ),
);

app.get("/weather", c => {
    return c.json({
        report: {
            weather: "sunny",
            temperature: 70,
        },
    });
});

serve({
    fetch: app.fetch,
    port: 4021,
});

console.log(`Server listening at http://localhost:4021`);
```

The `x402ResourceServer` talks to your facilitator and `paymentMiddleware` guards `/weather`. The `exact` scheme is the one implemented in x402 today; `$0.001` goes to your address on the network whose [CAIP-2](https://github.com/ChainAgnostic/CAIPs/blob/main/CAIPs/caip-2.md) id is `algorand:SGO1GKSzyE7IEPItTxCByw9x8FmnrCDexi9/cOUJOiI=` on Algorand TestNet.

Unpaid `GET /weather` returns **402** with requirements; paid calls return the JSON body.

This matches **Resource server setup** (`x402-demo` `server/hono`). Other server stacks live under [`server/`](https://github.com/algorand-devrel/x402-demo/tree/main/server).

#### How the resource server works

The resource server **defines** what the client must pay, **talks to the facilitator** to verify and settle, and only then returns data. Settlement on Algorand often uses atomic transaction groups (for example when another account sponsors fees).

### Part 3: Run the full example locally

#### Start the resource server

In one terminal, from the server project:

```bash
npx tsx index.ts
```

Listen on `http://localhost:4021` (or the port you configured). **Leave this server running** while you start the client in the next step.

#### Start the client

In a second terminal, open your **client** project’s **`index.ts`** and point **`url`** at your local **`/weather`** route (default port **4021**):

```typescript
// During Parts 1–2 you used:
const url = "https://x402.goplausible.xyz/examples/weather";

// For this local run, use instead:
const url = "http://localhost:4021/weather";
```

Run `npx tsx index.ts` again.

#### Note on the facilitator

The **facilitator** verifies payments and settles on behalf of the resource server. This walkthrough uses GoPlausible’s hosted service via **`FACILITATOR_URL`**.

When the client submits payment proof, **`HTTPFacilitatorClient`** and **`x402ResourceServer`** on your resource server call that endpoint; the server forwards the payload for verification and settlement on Algorand.

For a directory of other facilitators, see [**Learn more about x402 on Algorand**](#learn-more-about-x402-on-algorand) at the end of this page.

#### Observe the full payment flow

You should see the full sequence:

1. Client requests the protected resource.
2. Resource server responds with **402 Payment Required** and requirements.
3. Client prepares and submits payment (wrapped `fetch` + `@x402-avm`).
4. Facilitator verifies and settles on Algorand.
5. Resource server returns the **weather** JSON.

This final step confirms the full end-to-end x402 flow on Algorand.
